Phase 2 · Access

Email magic-link auth, signed session cookies, and a per-document access check that the WebSocket handshake reuses.

  2. Session cookie. Verify token, set httpOnly SameSite cookie.
  3. Shared ACL guard. One canAccess() for REST + socket upgrade.

Phase 3 · Core

Realtime sync engine — the hard part

A CRDT-based sync server that fans out document updates to every connected client and periodically persists snapshots. This is the phase currently in progress.

Build steps

  1. WebSocket room manager. One in-memory room per doc, keyed by id.
  2. Yjs update relay. Broadcast binary updates to all peers in the room.
  3. Debounced persistence. Flush the merged Y.Doc to ydoc_state every 2s of quiet.
  4. Presence & awareness. Cursors, selections, who's-online.
  5. Reconnection & offline replay. Buffer local updates, replay on reconnect.

Phase 4 · Frontend

Collaborative UI — the editor surface

A TipTap/ProseMirror editor bound to the shared Y.Doc, remote cursors with name labels, and a presence avatar stack.

Build steps

  1. Bind editor to Y.Doc. TipTap + y-prosemirror two-way binding.
  2. Remote cursors. Render awareness state as labelled overlays.
  3. Presence avatars. Who's-online stack in the toolbar.

Phase 5 · Operations

Deploy & observability — ship it, watch it

Containerize, deploy behind a sticky-session load balancer, and wire metrics so we can see room counts, latency, and snapshot lag in production.

Build steps

  1. Containerize. Dockerfile + /ready health check.
  2. Sticky sessions. LB cookie so a peer stays on one node.
  3. Dashboards. Prometheus metrics + Grafana panels.
