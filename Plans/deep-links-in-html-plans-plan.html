Phase 1 · Foundation

Schema + URL builder — the data + the encoder

Everything downstream depends on two things: a place to declare the deep-link config, and one JS helper that builds correctly-encoded URLs. Centralising URL construction in JS means encodeURIComponent handles all escaping and we never hand-encode in markup.

#plan-state — new deepLinks block (back-compatible, optional)
deepLinks.cwdstringabsolute repo path (Windows ok; no UNC)
deepLinks.planPathstringpath to this plan file, for "continue" prompts
deepLinks.sessionIdstring?optional — enables vscode:// same-session resume (Phase 4)
actions[id]{start,verify}per-phase prompt text, keyed by tracked id
  1. Add the deepLinks + actions reader to the IIFE, alongside the existing status/steps parse. Absent → feature silently off.
  2. Write ccUrl({q, cwd, repo, session, vscode}) — returns a claude-cli://open?… string (or vscode://anthropic.claude-code/open?… when vscode), URL-encoding q and joining params.
  3. Guard the 5,000-char limit — if an encoded q exceeds it, drop a console.warn and truncate-with-notice rather than emitting a broken link.
  4. Sanity-check encoding — newlines become %0A, quotes/ampersands survive a round-trip.
No visual change yet — this phase is pure plumbing. Phases 2–3 consume ccUrl().

Phase 3 · Centerpiece

Select & Answer — the plan answers its own questions

This is the most useful part. The skill already pauses by writing a waiting field; today you answer in the chat. Here, the waiting banner grows clickable option buttons and a free-form "Answer" button — so you reply from the doc, ideal for the second-monitor / async case the skill already targets.

  2. Compose the answer prompt — Re: "<question>" I choose: <label>. Continue the plan in <planPath>., encoded via ccUrl().
  3. Add free-form "Answer in Claude Code" — opens a session pre-filled Re: <question> — for the user to finish typing.
  4. Route same-session vs new — use vscode://…&session= when sessionId is set (Phase 4), else claude-cli:// new session that reloads context from planPath.
Honest scope: clicking an option does not answer a live AskUserQuestion modal — see The one-way constraint. It launches/resumes a session with the answer pre-filled; the user presses Enter. The win is the async case.
Phase 4 · Spike

Session-id capture — true same-session resume

The only path to resuming the same conversation (rather than a fresh one) is vscode://anthropic.claude-code/open?session=<id>&prompt=…. We run in the VSCode extension, so this is viable — if the skill can learn its own session id and write it into deepLinks.sessionId. That capture mechanism is the open question.

If capture fails
  • Fall back to claude-cli:// new session
  • Prompt says "read planPath & continue"
  • Always works — no id needed
If capture works
  • vscode://…&session=<id>
  • Resumes the same chat in-context
  • Closest to "answer the live question"
  1. Investigate id sources — env var, transcript path, or SDK-exposed session id available to the skill at generation time.
  2. If found: write deepLinks.sessionId; switch select/answer buttons to vscode://.
  3. If not: document the limitation; ship with the Phase 3 new-session fallback (graceful, no regression).
Time-boxed spike. The feature ships fully functional without it — this only upgrades the resume from "new session reloads context" to "same session continues".
Phase 5 · Docs

SKILL.md documentation — teach Claude to wire it

On-by-default means the template renders the buttons, but SKILL.md must tell Claude how to populate deepLinks + actions and write good prompts.

  1. New "Deep-link actions" section — the three primitives (Trigger / Select / Answer) and when each applies.
  2. cwd guidance — the skill runs inside the repo, so set deepLinks.cwd to the absolute working dir; prefer it over repo.
  3. Prompt-writing rules — ≤5,000 chars, point at the plan file for long work, encoding is automatic.
  4. Caveats, documented honestly — requires Claude Code v2.1.91+; handler registers on first claude run; deep-link-only means a click silently no-ops if unregistered; GitHub strips schemes but local file:// works.
Phase 6 · Proof

Verify end-to-end — click every button

  1. Generate a sample plan with deepLinks + actions populated, open in Chrome.
  2. Click each trigger — Open / Start / Verify / Continue — confirm Claude Code opens in the right cwd with the right prompt.
  3. Set a waiting with options — confirm option buttons + "Answer" render and launch with the choice pre-filled.
  4. Regression check — a plan with no deepLinks renders exactly as before (no buttons, no errors).
Acceptance: all four button types fire correctly, the async answer loop works, and deep-link-free plans are unchanged.

Reality check

The one-way constraint

A deep link can only open or resume a session with a prompt pre-filled. It cannot inject into a running turn. When Claude is blocked on a live AskUserQuestion, it is waiting on the harness — a browser click cannot reach into that turn.

✗ Not possible
  • Click answers a live modal in place
  • Two-way back-channel into a running turn
  • Auto-send without pressing Enter
✓ What we build
  • Async: Claude poses Q in waiting, ends turn
  • User clicks later → session opens pre-filled
  • Press Enter → continues, reloads context
The doc states this plainly so nobody expects magic. The async select/answer flow is the genuine win — and it maps onto the waiting + 🔔 notify machinery that already exists.
