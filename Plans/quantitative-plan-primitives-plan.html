Quantitative Plan Primitives

Why

Context & rationale

The template's diagram primitives (.flow, .schema, .tree, .steps, .compare) cover qualitative structure. They can't show quantitative plan content — and that content currently gets buried in prose or tables.

Today — quantitative gaps
  • Time — which phases overlap, what blocks what, critical path
  • Magnitude — relative effort per phase; a before/after metric
  • Composition — how a whole splits ("60% is the migration")
After — three primitives
  • .gantt → time / overlap / sequencing
  • .hbar → magnitude (effort, or before/after)
  • .stack → composition of a whole
The constraint that shapes everything: the page does a hard location.reload() every 4 s (plan-template.html:1187). Anything that lays out in JS after first paint re-flows and flickers each reload. So every primitive is static markup with author-baked numbers — Claude computes the widths / column spans at write time; the browser just paints styled elements. No runtime JS, no libraries.
Scope

Scope & cuts (YAGNI)

Five primitives were considered; three are in scope. The cuts are deliberate and recorded so they aren't relitigated.

PrimitiveDecisionReason
.gantt✅ BuildThe one thing nothing else shows — overlap, sequencing, critical path.
.hbar✅ BuildTwo jobs: per-phase effort and before/after metric (perf/migration plans).
.stack✅ BuildComposition of a whole — distinct from .hbar, nearly free.
.spark⊘ CutNeeds a time-series most plans lack; the progress ring already shows "how far along."
.matrix⊘ CutOnly useful when a plan is a prioritization exercise. Revisit on real need.
.dag⊘ CutRare at plan scale; .flow/.gantt/.tree cover it. A runtime lib (dagre) would be the only ~100 KB inline dep and the only one needing a flicker workaround — bad trade.
Phase 1 · Magnitude

.hbar — labelled bars: effort, or before/after

Start here — it's the simplest and proves the static-markup model. A standalone block (no .diagram wrapper, like .compare): rows of a left label, a track, and a .bar sized by inline width:%, with the value at the bar's end.

  1. Add CSS to <style>. .hbar (rows grid), .hbar .row, .lbl, .track, .bar; colors from --accent / --muted; the value label inline at bar end.
  2. Single-series variant. One bar per row = per-phase effort/size. Author writes style="width:62%" as a fraction of the row max.
  3. Two-series variant. .bar.before (muted) + .bar.after (accent) stacked in one row, so shrink/growth reads at a glance.
  4. Verify in isolation. Render both variants; check light/dark + narrow viewport before moving on.
Authoring contract: widths are percentages of the row's own max value, baked in by the author. No JS reads the data.
Phase 2 · Composition

.stack — a 100% bar split into segments

One horizontal bar whose children are tinted segments. Uses flexbox so the author writes raw values as flex:<n> — flexbox normalizes to proportions, so no percentage math is needed.

  1. Add CSS. .stack (flex row, rounded, overflow-hidden), .stack .seg tinted from the accent palette (--accent, --accent-2, --cyan, --green, --amber), plus a .legend row mapping colour → label → value.
  2. Segment authoring. Each .seg style="flex:60"; small segments still show a sliver. Document a 2–6 segment sweet spot — beyond that, use a table.
  3. Legend. Reuse the existing .legend styling so it themes consistently.
  4. Verify. 3-segment example, light/dark, narrow viewport.
Phase 3 · Time

.gantt — phases across time; the most complex

Wrapped in the standard .diagram + .dg-title. A CSS grid with a leading label column plus N time columns; each task is a .gbar placed by an inline grid-column: start / end. Overlap and sequencing become visible because bars share the same column axis.

  1. Add CSS. .gantt (grid, grid-template-columns = label + repeat(N)), header row of column labels, .gtask label cells, .gbar with rounded fill + inline duration text. Optional data-accent to mark the critical path.
  2. Placement convention. Author writes style="grid-column: 2 / 5" per bar. Document the "column = time unit" rule and a worked 5-task example with one overlap.
  3. Overflow. Wrap in an overflow-x:auto container so a wide chart scrolls inside its own box and never causes horizontal page scroll.
  4. Verify. 5 tasks incl. one overlap + one critical-path tint; light/dark; narrow viewport scrolls within the box.
Watch: the gantt is the one most likely to push page width. The overflow-x:auto wrapper is mandatory, matching how other wide content is handled.
Phase 4 · Docs

Documentation — make the primitives discoverable

Three coordinated doc edits so future plans actually use these — and use them appropriately.

  1. Cheatsheet comment. Add a "Quantitative" group to the top-of-file comment (plan-template.html:21-32), one line each.
  2. In-body example patterns. Drop a copy-paste-ready commented block per primitive near the existing examples (~:515-560), with realistic baked numbers.
  3. SKILL.md table. Add the three to the Diagrams table (SKILL.md:58-66) with a one-line when to use and the honest when not to (the YAGNI filter), so plans don't sprout charts gratuitously.
Phase 5 · Proof & rollout

Verify & sync — visual proof, then propagate

This is presentation markup, so verification is visual; then propagate the change to the two mirrors.

  1. Build a throwaway exercise plan using all three (single + dual .hbar, a 5-task .gantt with overlap, a 3-segment .stack).
  2. Open & observe in the plans Chrome instance: renders in light and dark; no flicker across ≥2 reloads; no horizontal page scroll; holds at a narrow viewport; existing primitives + tracker unaffected.
  3. Sync mirrors. Copy to the installed ~/.claude/skills/rich-html-plans/ and port to the Quadshift/ai-library mirror.
Done when: all three render correctly in both themes with no flicker / no page-scroll, docs updated, and both mirrors carry the change.
